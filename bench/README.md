# Synthetic GPU Benchmark Suite

A controllable benchmark suite that generates GPU profiles with known, reproducible
performance bottlenecks. Two implementations target different clusters:

| Variant | Compiler | Profiler | Cluster | Subdir |
|---------|----------|----------|---------|--------|
| CUDA | nvcc | Nsight Systems (`.nsys-rep` → `.sqlite`) | Perlmutter (NERSC, A100) | `cuda/` |
| HIP  | hipcc | rocprof-sys (`.rocpd`)                   | Frontier (OLCF, MI250X)  | `hip/`  |

Ground-truth JSON files (one per run, written by rank 0) record the actual scenario
name and expected bottleneck label for offline comparison against advisor output.
`ground_truth_meta.json` (this directory) is shared by both variants — the scenario
names and evaluation rubric are identical across CUDA and HIP.

## How to Use — CUDA (Perlmutter)

**1. Build** (from `bench/cuda/` on a Perlmutter login node):

```bash
cd bench/cuda
bash build.sh
```

Produces `synthetic_cuda_benchmark_mpi`. Loads `cudatoolkit`, detects the Cray PE
GTL library automatically, and compiles for `sm_80` (A100). Use `--arch sm_90` for
H100, `--debug` for Nsight source-level correlation.

**2. Submit** from `bench/cuda/` (scripts use `pwd` to locate the binary):

```bash
cd bench/cuda
sbatch submit_1gpu.sbatch   # tests #1–#4:  single-GPU
sbatch submit_4gpu.sbatch   # tests #5–#7:  intra-node multi-GPU
sbatch submit_8gpu.sbatch   # test  #8:     inter-node MPI
```

**3. Outputs** (written to `bench/cuda/profiles/{1gpu,4gpu,8gpu}/`):

- `<id>.nsys-rep` / `<id>.<rank>.nsys-rep` — Nsight Systems profile
- `<id>.sqlite`   / `<id>.<rank>.sqlite`   — SQLite export for PerfAdvisor
- `<id>.json`                               — ground-truth: scenario, bottleneck, params

---

## How to Use — HIP (Frontier)

**1. Build** (from `bench/hip/` on a Frontier login node):

```bash
cd bench/hip
bash build_hip.sh
```

Produces `synthetic_hip_benchmark_mpi`. Loads `PrgEnv-amd`, `rocm`,
`craype-accel-amd-gfx90a`, detects GTL. Use `--arch gfx942` for MI300X.

**2. Submit** from `bench/hip/` (scripts use `SLURM_SUBMIT_DIR`):

```bash
cd bench/hip
sbatch submit_1gpu_hip.sbatch    # tests #1–#4:  single-GCD
sbatch submit_4gpu_hip.sbatch    # tests #5–#7 + #10: intra-node multi-GCD
sbatch submit_8gpu_hip.sbatch    # test  #8:     inter-node MPI
```

**3. Outputs** (written to `bench/hip/profiles/{1gpu,4gpu,8gpu}/`):

- `<id>/rank_<pid>.rocpd` — rocpd SQLite files (one per rank, in a per-run subdirectory)
- `<id>.json`             — ground-truth: scenario, bottleneck, params

No export step needed — `.rocpd` files are already SQLite databases consumed directly
by PerfAdvisor's rocpd ingestion path.

## Run Reference

### Single-GPU / single-GCD runs → `profiles/1gpu/`

| Test | Scenario | Bottleneck | Key params |
|------|----------|------------|------------|
| test_01 | scen_a | kernel launch overhead | 10 K tiny launches |
| test_02 | scen_f | CPU sync stall | sync after every launch |
| test_03 | scen_b | PCIe transfer-bound | 16 chunks, transfer > compute |
| test_04 | scen_c | transfer-compute overlap missing | same as test_03, serialized |

### Intra-node multi-GPU / multi-GCD runs → `profiles/4gpu/`

| Test | Scenario | Bottleneck | Key params |
|------|----------|------------|------------|
| test_05 | scen_i | unnecessary host staging (P2P) | D2H → MPI_Sendrecv → H2D, 4 ranks |
| test_06 | scen_j | MPI load imbalance | skewed work-iters × rank |
| test_07 | scen_k | host-staged collective | D2H → MPI_Allreduce → H2D |
| test_10 | scen_h | *(HIP only)* xGMI optimal path | ROCm-aware MPI, device ptrs — excluded from accuracy tallies |

### Inter-node MPI runs → `profiles/8gpu/`

| Test | Scenario | Bottleneck | Key params |
|------|----------|------------|------------|
| test_08 | scen_l | host-staged halo exchange | 8-rank ring, D2H → MPI_Sendrecv → H2D |

---

**test_01 — kernel launch overhead**

10,000 back-to-back launches of a tiny kernel (16 K elements, 2 inner ops each).
The GPU is almost never busy; the timeline is dominated by launch latency gaps.

*Resolution:* Fuse work into fewer, larger kernels. Use CUDA Graphs / HIP Graphs to
record and replay launch sequences with near-zero CPU overhead.

---

**test_02 — CPU sync stall**

2,000 kernel launches each followed immediately by `cudaStreamSynchronize` /
`hipStreamSynchronize`. The CPU blocks after every launch, preventing any overlap.

*Resolution:* Move synchronization to stage boundaries. Use CUDA/HIP events to chain
dependent work without stalling the CPU.

---

**test_03 — PCIe transfer-bound**

Large H2D → compute → D2H pipeline across 16 chunks. Each chunk's compute stage
(~25 µs) is dwarfed by PCIe transfers (~250 µs each way).

*Resolution:* Increase arithmetic intensity. Keep data GPU-resident across iterations
to eliminate the round-trip H2D/D2H.

---

**test_04 — transfer-compute overlap missing**

Identical workload to test_03 but all streams are flushed after each chunk, making
the pipeline strictly sequential.

*Resolution:* Issue `cudaMemcpyAsync` / `hipMemcpyAsync` on a transfer stream while
a separate compute stream processes the previous chunk.

---

**test_05 — unnecessary host staging, intra-node P2P**

4-rank ring `MPI_Sendrecv` with 64 MB halo buffers, explicit D2H → MPI → H2D
staging on every transfer, even though all GPUs are on the same node.

*Resolution (CUDA):* Use CUDA-aware MPI (`--cuda-aware-mpi 1`). *Resolution (HIP):*
Use ROCm-aware MPI (`--rocm-aware-mpi 1`) so the MPI library transfers directly
between device buffers via xGMI (Infinity Fabric).

---

**test_06 — MPI load imbalance (barrier stall)**

4 ranks perform SFU-heavy compute where rank *k* does `work_iters × (k+1)`
iterations. All ranks call `MPI_Barrier`. Fast ranks idle at the barrier.

*Resolution:* Balance work across ranks. For inherently uneven workloads, use
asynchronous collectives (`MPI_Ibarrier`) so fast ranks can queue follow-on work.

---

**test_07 — host-staged collective (MPI_Allreduce)**

4 ranks each run one kernel then D2H → `MPI_Allreduce` → H2D. Every reduction
round-trips through host memory.

*Resolution (CUDA):* CUDA-aware `MPI_Allreduce` or NCCL `ncclAllReduce`.
*Resolution (HIP):* ROCm-aware `MPI_Allreduce` or RCCL `rcclAllReduce`.

---

**test_08 — inter-node halo exchange, host-staged**

8-rank ring `MPI_Sendrecv` with 64 MB halos, host-staged (D2H → MPI → H2D) across
two nodes (all ring neighbors cross the network).

*Resolution:* GPU-Direct RDMA (CUDA-aware / ROCm-aware MPI with GDRCopy) to remove
staging copies. Or, if unavailable, double-buffering to overlap staging with compute.

---

**test_10 — xGMI optimal path** *(HIP / Frontier only)*

4-rank ring `MPI_Sendrecv` with device pointers and ROCm-aware MPI enabled. All
transfers stay on-chip via the xGMI Infinity Fabric links between GCDs. This is the
optimal reference baseline with no injected deficiency; it is excluded from detection
accuracy tallies.

## Evaluating PerfAdvisor

`ground_truth_meta.json` (this directory) holds the static evaluation rubric — one
entry per scenario, keyed by the `scenario` field in the runtime ground-truth JSONs.

**Evaluate CUDA profiles (Perlmutter):**

```bash
python -m perf_advisor evaluate bench/cuda/profiles/ \
    --ground-truth bench/ \
    --model claude-opus-4-6 \
    --output eval_results_cuda.json
```

**Evaluate HIP profiles (Frontier):**

```bash
python -m perf_advisor evaluate bench/hip/profiles/ \
    --ground-truth bench/ \
    --model claude-opus-4-6 \
    --output eval_results_hip.json
```

**Rescore without re-running PerfAdvisor** (after editing the rubric or judge prompt):

```bash
python -m perf_advisor evaluate \
    --cached eval_results_cuda.json \
    --ground-truth bench/
```

**Bottleneck-detection-only check** (no judge API calls, fast):

```bash
python -m perf_advisor evaluate bench/cuda/profiles/ \
    --ground-truth bench/ \
    --skip-judge --yes
```

**Per-suggestion detail** (shows judge score and explanation for every suggestion):

```bash
python -m perf_advisor evaluate bench/cuda/profiles/ \
    --ground-truth bench/ \
    --output eval_results_cuda.json \
    --verbose
```

### Scoring

| Axis                 | Method                                                               | Output        |
| -------------------- | -------------------------------------------------------------------- | ------------- |
| Bottleneck detection | Enum map + keyword fallback; deterministic                           | ✓ / ✗ per run |
| Suggestion coverage  | LLM judge (default: Haiku); scores 0 / 1 / 2 per expected suggestion | raw score + % |
| False positives      | Count hypotheses with no match to expected bottleneck                | count per run |

The judge model defaults to `claude-haiku-4-5-20251001` regardless of the hypothesis
model. Override with `--judge-model`. Optimal-path runs (test_10) are shown in the
results table but excluded from detection accuracy and coverage tallies.

Key observations:

- Opus-4-6 leads on detection rate (8/8) and suggestion coverage (62.5%), though
  false positives are slightly higher than gpt-5.4.
- GPT-5.4 is close runner-up — 8/8 detection, 54.2% coverage, lowest false positives
  among the 8/8 group (2.50 avg).
- Haiku-4-5 matches on detection but has the most false positives (3.0 avg) and drops
  to 50% coverage.
- GPT-4o misses two scenarios entirely (sync_stall, overlap_missing) and coverage
  drops sharply on multi-GPU MPI tests.
- Gemini-2.5-flash was partially functional — 2 API errors, detected 5/8, weak
  coverage (0% on both 4-GPU MPI tests).
- Gemini-3.1-pro-preview was fully down — all 8 runs hit 503 UNAVAILABLE.

Common weakness: MPI-specific suggestions (CUDA/ROCm-aware MPI, GPU-Direct RDMA,
NCCL/RCCL, peer access) are consistently missed or only partially covered. Single-GPU
kernel scenarios score better than multi-GPU MPI tests across all models.
