# Synthetic GPU Benchmark Suite

A controllable benchmark suite that generates GPU profiles with known, reproducible
performance bottlenecks. Two implementations target different clusters:

| Variant | Compiler | Profiler | Cluster | Subdir |
|---------|----------|----------|---------|--------|
| CUDA | nvcc | Nsight Systems (`.nsys-rep` → `.sqlite`) | Perlmutter (NERSC, A100) | `cuda/` |
| HIP  | hipcc | rocprof-sys (rocpd `.db`)                | Frontier (OLCF, MI250X)  | `hip/`  |

Ground-truth JSON files (one per run, written by rank 0) record the actual scenario
name and expected bottleneck label for offline comparison against advisor output.
`ground_truth_meta.json` (this directory) is shared by both variants — the scenario
names and evaluation rubric are identical across CUDA and HIP.

## How to Use — CUDA (Perlmutter)

**1. Build** (from `bench/cuda/` on a Perlmutter login node):

```bash
cd bench/cuda
bash build.sh
```

Produces `synthetic_cuda_benchmark_mpi`. Loads `cudatoolkit`, detects the Cray PE
GTL library automatically, and compiles for `sm_80` (A100). Use `--arch sm_90` for
H100, `--debug` for Nsight source-level correlation.

**2. Submit** from `bench/cuda/` (scripts use `pwd` to locate the binary):

```bash
cd bench/cuda
sbatch submit_1gpu.sbatch   # tests #1–#4:  single-GPU
sbatch submit_4gpu.sbatch   # tests #5–#7:  intra-node multi-GPU
sbatch submit_8gpu.sbatch   # test  #8:     inter-node MPI
```

> [!IMPORTANT]
> **Run from a parallel filesystem, not a home directory.** Profiles are written
> next to the submit directory, and Nsight Systems cannot create a `.nsys-rep` on a
> home filesystem: the run completes, a `.qdstrm` appears, the conversion fails with
> `errno 524` (ENOTSUPP), and **nsys still exits 0** — so the job looks successful
> until the export step reports the file missing. Put the checkout on scratch
> (`$SCRATCH` / `$PSCRATCH` on NERSC systems, the project Lustre space on OLCF) and
> submit from there. rocpd files are SQLite databases, so the same advice applies to
> the HIP runs.

**3. Outputs** (written to `bench/cuda/profiles/{1gpu,4gpu,8gpu}/`):

- `<id>.nsys-rep` / `<id>.<rank>.nsys-rep` — Nsight Systems profile
- `<id>.sqlite`   / `<id>.<rank>.sqlite`   — SQLite export for PerfAdvisor
- `<id>.json`                               — ground-truth: scenario, bottleneck, params

## How to Use — HIP (Frontier)

**1. Build** (from `bench/hip/` on a Frontier login node):

```bash
cd bench/hip
bash build_hip.sh
```

Produces `synthetic_hip_benchmark_mpi`. Loads `PrgEnv-amd`, `rocm`,
`craype-accel-amd-gfx90a`, detects GTL. Use `--arch gfx942` for MI300X.

**2. Submit** from `bench/hip/` (scripts use `SLURM_SUBMIT_DIR`):

```bash
cd bench/hip
sbatch submit_1gpu_hip.sbatch    # tests #1–#4:  single-GCD
sbatch submit_4gpu_hip.sbatch    # tests #5–#7:  intra-node multi-GCD
sbatch submit_8gpu_hip.sbatch    # test  #8:     inter-node MPI
```

**3. Outputs** (written to `bench/hip/profiles/{1gpu,4gpu,8gpu}/`):

- `<id>.db` (single-rank) or `<id>.<rank>.db` (multi-rank) — rocpd SQLite files
- `<id>.json`             — ground-truth: scenario, bottleneck, params

No export step needed — rocpd `.db` files are already SQLite databases consumed
directly by PerfAdvisor's rocpd ingestion path.

**Layout matches the CUDA side.** rocprof-sys natively writes a per-run
subdirectory (`<id>/[rank_]rocpd-<N>.db`); the submit scripts flatten it to
`<id>.<rank>.db` after each run so both benches produce the same shape and
`discover.py` needs one convention. Flattening is a post-step rather than a
rocprof-sys output-variable change because the rename is deterministic and, if it
finds nothing, leaves the original files untouched instead of writing them
somewhere unexpected. Captures taken before this change still load — the per-run
subdirectory remains a supported layout.

## Run Reference

> [!IMPORTANT]
> **This section is the canonical record of what each test exists to trigger, and
> why.** Read it before changing anything under `bench/` or `perf_advisor/eval/`,
> and before concluding that any test is redundant — the code cannot state why a
> scenario exists, and the ground-truth JSONs record only labels. A test whose
> purpose you cannot find here has an undocumented purpose, not an absent one:
> establish it and write it down before touching the test. Update this section in
> the same change as the code.
>
> The table below is a mechanical index (what runs, with what arguments, and
> whether a profile exists). **The per-test descriptions further down carry the
> intent** — what bottleneck is being provoked and how it shows up in a profile.

Reading the table:

- **Scenario** — the opaque `scen_*` ID passed to `--scenario`, plus its internal
  name. The IDs are deliberately non-descriptive: profilers capture argv, so a
  descriptive name would hand the advisor the answer.
- **Runner** — the C++ function that implements it. Two tests sharing a runner
  produce structurally similar profiles; that is worth knowing before treating
  them as independent evidence.
- **Captured** — whether a profile currently exists for the eval to score, per
  platform.

Every row was verified against both sources and all six sbatch scripts. Per-test
intent is in [Test descriptions](#test-descriptions) below.

### The scenario table (both platforms)

**One table, because the two suites are ports of one another.** Scenario IDs,
runner functions and benchmark arguments are identical on CUDA and HIP for every
row — a divergence in any of those is a bug in one of them (CLAUDE.md, "Two
benchmarks, kept in lockstep"). Keeping a single table means such a divergence
has to be written into a cell to exist, rather than hiding between two lists that
nobody diffs.

Platform differences are confined to the last two columns and are the only ones
that should ever appear there: GPU/GCD vocabulary, the Slurm binding flag, and
which captures currently exist.

## Test descriptions

What each test is *for*: the bottleneck it provokes and how that bottleneck
presents in a profile. This is the record to consult before assuming anything
about a test's intent, and before removing one. Each block states the **expected bottleneck** (the `expected_bottleneck` label in
the run's ground-truth JSON, plus the advisor `bottleneck_type` enum(s) the scorer
accepts for it) and the **expected fixes**.

The expected-fix lists mirror `ground_truth_meta.json` → `<scenario>.suggestions`,
in rubric order and one line per entry. **`ground_truth_meta.json` remains
authoritative** — the judge scores against its `action` and `mechanism` text, not
against these summaries. They are here so a reader can see what a correct answer
looks like without opening the JSON; if you change one, change both.

Only `kernel_a`, `kernel_b` and `kernel_d` exist — there is no `kernel_c`. `kernel_a`
is reused by `scen_a`, `scen_k`, and the halo consume step in `scen_i` / `scen_l`.

**test_01 (`scen_a` tiny_kernels) — kernel launch overhead**

*Expected bottleneck:* `kernel_launch_overhead` — advisor enum `cpu_launch_overhead`.

10,000 back-to-back launches of a tiny kernel (16 K elements, 2 inner ops each).
The GPU is almost never busy; the timeline is dominated by launch latency gaps.

`--host-sleep-us 5` additionally forces a `cudaStreamSynchronize(0)` /
`hipStreamSynchronize(0)` after *every* launch — the same per-launch stall test_02
injects deliberately. This is load-bearing, not an oversight: without it the
launches pipeline and the launch latency stops being visible. It does mean test_01
carries test_02's bottleneck as a genuine secondary observation, which is why
`tiny_kernels` declares a `synchronization` entry in `also_true`.

*Expected fixes:*
1. CUDA/HIP Graphs — record the launch sequence once and replay it with near-zero
   per-launch CPU overhead (`cudaGraphCreate` / `Instantiate` / `Launch`).
2. Fuse several small kernels into one larger kernel so each launch does more work.
3. Collapse the repeated launches into a single persistent kernel that loops
   internally over the iterations (grid-stride or internal loop) instead of
   relaunching, removing the per-launch CPU cost.

---

**test_02 (`scen_f` sync_stall) — CPU sync stall**

*Expected bottleneck:* `cpu_sync_stall` — advisor enum `synchronization`.

2,000 kernel launches each followed immediately by `cudaStreamSynchronize` /
`hipStreamSynchronize`. The CPU blocks after every launch, preventing any overlap.

*Expected fixes:*
1. Remove the per-launch `cudaStreamSynchronize`; synchronise only at genuine
   data-dependency boundaries (e.g. before a CPU read of GPU-produced data).
2. Express any real dependency with an event rather than a blocking synchronize —
   `cudaEventRecord` on the producing work, wait on it from the dependent work —
   so the CPU thread is not stalled. (The scenario is single-stream, so this is
   about not blocking the CPU, not about inter-stream ordering.)
3. If the CPU genuinely needs results, batch the synchronisation at the end of a
   full pipeline stage rather than after each kernel.

---

**test_03 (`scen_b` transfer_bound) — PCIe transfer-bound**

*Expected bottleneck:* `pcie_transfer_bound` — advisor enum `memory_bound` or `io`.

Large H2D → compute → D2H pipeline across 16 chunks on 4 streams. Each chunk's
compute stage is dwarfed by PCIe transfers, so the pipeline is transfer-bound.

> Measured from `test_03.sqlite` (320 chunks = 20 reps × 16): per 8.4 MB chunk,
> compute (`kernel_b`) ~68 µs vs H2D ~403 µs and D2H ~390 µs — transfer ≫ compute
> by ~6×, which is the structural ordering the scenario depends on.

*Expected fixes:*
1. Raise arithmetic intensity — significantly more computation per byte
   transferred, so GPU time dominates transfer time.
2. Keep data GPU-resident across iterations to eliminate the H2D/D2H round trip
   entirely; allocate outputs on the device and transfer only final results back.
3. Reduce the bytes crossing PCIe: transfer in a lower precision (fp16/bf16) and
   upconvert on the device, or compress the payload before the copy.

The transfers are *already* pinned (`cudaMallocHost`), async (`cudaMemcpyAsync`)
and overlapped across 4 streams, so the cheap staging optimisations are done — the
only remaining transfer lever is moving less data (fix 3), and everything else is
algorithmic (fixes 1–2).

---

**test_04 (`scen_c` overlap_missing) — transfer-compute overlap missing**

*Expected bottleneck:* `cpu_gpu_overlap_missing` — advisor enum `synchronization` or `io`.

Identical workload to test_03 but all streams are flushed after each chunk, making
the pipeline strictly sequential.

*Expected fixes:*
1. Pipeline chunks across streams — while stream A computes chunk K, stream B
   transfers K+1 H2D and stream C transfers K-1 D2H (at least 3 streams, all
   operations issued asynchronously).
2. Remove the per-chunk stream flush that serialises the pipeline, so chunk K+1's
   H2D can begin before chunk K's compute finishes; express the dependency with
   `cudaEventRecord` / `cudaStreamWaitEvent` instead of a synchronise.
3. Order chunks with events rather than a coarse per-chunk flush, and keep several
   chunks in flight so a stall on one does not idle the others (deeper pipelining).

Item 2 is the scenario's own defect — the flush is what `serialized=true` injects —
so a hypothesis that recommends only generic async-copy advice has not diagnosed
this profile. Note the copies are *already* `cudaMemcpyAsync` on page-locked
buffers; the flush is what defeats them.

---

**test_05 (`scen_i` p2p_staged) — unnecessary host staging, intra-node P2P**

*Expected bottleneck:* `unnecessary_host_staging_intranode` — advisor enum `mpi_latency`.

4-rank ring `MPI_Sendrecv` with 64 MB halo buffers, explicit D2H → MPI → H2D
staging on every transfer, even though all GPUs are on the same node. A small
kernel consumes each received halo, so the profile shows the low-but-non-zero GPU
utilisation of a real exchange rather than a pure transfer microbenchmark.

*Expected fixes:*
1. Pass device pointers directly to `MPI_Sendrecv` (CUDA-/ROCm-aware MPI) and drop
   the explicit D2H/H2D copies; requires `MPICH_GPU_SUPPORT_ENABLED=1`.
2. For NVLink/xGMI-connected GPUs on one node, enable peer access so MPI can use
   direct device-to-device copies (`cudaDeviceEnablePeerAccess` between rank pairs).
3. If CUDA-aware MPI is unavailable, bypass MPI entirely — share device pointers
   across processes with `cudaIpcGetMemHandle` / `cudaIpcOpenMemHandle` and exchange
   with `cudaMemcpyPeerAsync`.

All three turn on the ranks being *co-located*; that is what separates this from
test_08, and it is what the scorer's keyword list for this scenario encodes. The
profile answers it directly — all 4 ranks report the same host, so the cross-rank
pre-seed carries `num_nodes: 1` and `neighbor_ranks_colocated: true`.

The MPI *timing* must not be used for locality — it misleads. A blocking ring
`MPI_Sendrecv` buffers the send and blocks on the receive, so the long event measures
how long the *neighbour* took to arrive (rank skew and host-memory contention), not
link bandwidth. Measured across all 12 rank profiles, mean `MPI_P2P_EVENTS` duration
is 4.3–4.6 ms on this intra-node exchange versus 1.7–1.8 ms on test_08's inter-node
one — the co-located case reads ~2.5× *slower*. Locality comes from the hostnames,
not the timing.

---

**test_06 (`scen_j` mpi_barrier_stall) — MPI load imbalance (barrier stall)**

*Expected bottleneck:* `mpi_load_imbalance` — advisor enum `mpi_imbalance`.

4 ranks perform SFU-heavy compute (`kernel_d`) where rank *k* does
`work_iters × (k+1)` iterations over its own private arrays. All ranks then call
`MPI_Barrier`, so the faster ranks idle there. The cause shows in the trace as a
per-rank kernel-duration spread — `kernel_d` runs 18.2, 36.3, 54.3 and 72.2 ms on
ranks 0-3, a clean 1:2:3:4 ladder at `--work-iters 64`. The fastest rank (0) then
spends 55 of its 86 ms (64%) waiting in `MPI_Barrier`.

The imbalance is **GPU compute only** — there is deliberately no CPU-side stagger, so
the single mechanism in the trace is the per-rank GPU kernel-duration spread, not a
`nanosleep` or other host stall.

*Expected fixes:*
1. Balance work across ranks so each spends the same time in GPU compute before the
   barrier (equalise `work_iters`, or partition the domain proportionally). This is
   the correct and complete diagnosis.
2. Replace `MPI_Barrier` with `MPI_Ibarrier` so fast ranks can queue follow-on work
   while waiting. *(Valid for real code, but this benchmark cannot demonstrate the
   gain: `barrier_sync()` at the top of the next rep is itself an `MPI_Barrier`, so a
   fast rank blocks immediately — see `todo_list.md` #13, hoisting that barrier.)*
3. The ranks exchange no data, so the barrier enforces no real dependency — remove
   it, or if it marks a genuine step boundary, coarsen it to one barrier every N
   iterations. **Not** point-to-point sync between dependent pairs: there are no
   dependent pairs to synchronise.

Note the profile shows 40 `MPI_Barrier` calls where only 20 are the injected one —
`barrier_sync()` is also an `MPI_Barrier`, name-identical to the bottleneck. It is a
harness artefact, excused via `also_true`; hoisting it out of the timed loop is
deferred (`todo_list.md` #13).

---

**test_07 (`scen_k` mpi_allreduce) — host-staged collective (MPI_Allreduce)**

*Expected bottleneck:* `host_staged_collective` — advisor enum `mpi_latency`.

4 ranks each run one `kernel_a` then D2H → `MPI_Allreduce` → H2D over an 8 MB
vector (`--n 2097152` floats). Every reduction round-trips through host memory.

**Proportion, measured (rank 0):** kernel 0.3 ms · staging (D2H+H2D) ~23 ms ·
`MPI_Allreduce` 156 ms per 30 reps. The injected host staging is real and
detectable (the `D2H → collective → H2D` triplet is right there in the trace) but
is only ~13% of runtime — the network collective dominates. This is inherent to a
vector allreduce on this hardware (staging is PCIe-bound, the collective is
network-bound) and is not fought with parameters: scaling `--n` grows both
proportionally. It means fix 1's *speedup* is modest even though its *diagnosis*
is correct, and the highest-value fix is actually fix 3. Detection is unaffected —
`allreduce` discriminates this scenario from the other two MPI ones regardless of
whether the advisor leads with staging or with the collective's cost.

*Expected fixes:*
1. Pass device pointers directly to `MPI_Allreduce` (CUDA-/ROCm-aware MPI) to
   eliminate the D2H/H2D staging.
2. Use NCCL `ncclAllReduce` / RCCL `rcclAllReduce` — higher-bandwidth intra-node
   reductions over NVLink/xGMI and network-aware inter-node reductions, without
   involving the CPU.
3. If the reduced result is not consumed immediately, issue the collective
   non-blocking (`MPI_Iallreduce`) and overlap it with independent GPU work rather
   than blocking on it. Applicable here — the allreduce output is never read before
   the next iteration's kernel, so the reduction *could* overlap it — and it is the
   highest-value fix, since the collective is the dominant cost. *(The benchmark
   issues them serially, so the profile shows the missed overlap; the loop is not
   restructured to realise it.)*

---

**test_08 (`scen_l` mpi_halo_exchange) — inter-node halo exchange, host-staged**

*Expected bottleneck:* `host_staged_halo_exchange` — advisor enum `mpi_latency`.

8-rank ring `MPI_Sendrecv` with 64 MB halos, host-staged (D2H → MPI → H2D) across
two nodes (all ring neighbors cross the network). A small kernel consumes each
received halo so the GPU timeline is not empty.

Each rep also updates an **interior domain** (`--interior-mb`) that does
*not* depend on the received halo — it could have been issued before the exchange
and waited on afterwards, but runs strictly after it, leaving the exchange latency
fully exposed. That unrealized overlap is what expected fix 2 addresses. Without an
interior domain that fix names structure the profile does not contain, and an
advisor correctly reporting "there is no interior work here" would score *worse*
than one reciting textbook halo advice without reading the profile — inverting the
property the eval rests on. Reporting the missed overlap is therefore factual and
is excused via `also_true`; it is not the injected bottleneck, which remains the
host staging.

> **Interior sizing.** The overlap fix is only worth recommending when the interior
> is a meaningful 15-35% of exchange time. Measured at `--interior-mb 512`:
> interior/exchange = 24.4-25.4% (CUDA) / 18.0-25.3% (HIP) — in band. And eliminating
> the host staging still saves more than perfect overlap would (rank 0: CUDA staging
> 44% of span vs interior 19%; HIP 8% vs 5%), so `host_staged_halo_exchange` remains
> the right label. The interior kernel is bandwidth-bound at `--interior-iters 64`, so
> interior time scales ~linearly with `--interior-mb`; scale `mb`, not `iters` (iters
> would cross the memory→compute ridge).

*Expected fixes:*
1. CUDA-aware MPI with GPU-Direct RDMA, so the Slingshot NIC reads/writes device
   memory directly with no host staging (`MPICH_GPU_SUPPORT_ENABLED=1`,
   `MPICH_RDMA_ENABLED_CUDA=1`; GTL uses GDRCopy for the RDMA path).
2. If GPU-Direct RDMA is unavailable, cover the cross-node latency with *interior
   compute*: post `MPI_Irecv`/`MPI_Isend`, launch the interior kernel, then process
   the boundary once the neighbour data lands.
3. Reduce halo size or exchange frequency by raising the subdomain volume-to-surface
   ratio.

The cross-node hop is visible in the profile without inferring it from timing: the
8 ranks report two alternating hosts, so the pre-seed carries `num_nodes: 2` and
`neighbor_ranks_colocated: false` — `--distribution=cyclic` placing every ring
neighbour on the far node. Counter-intuitively this run's MPI calls are *faster* than
test_05's — a blocking ring `Sendrecv` measures neighbour skew, not bandwidth (see
test_05) — so locality is read from the hostnames, not the timing.

Item 2 is deliberately the interior/boundary split, not generic double-buffering — a
mechanism specific to an inter-node halo, so it is not satisfiable by another
scenario's correct answer.

## Evaluating PerfAdvisor

`ground_truth_meta.json` (this directory) holds the static evaluation rubric — one
entry per scenario, keyed by the `scenario` field in the runtime ground-truth JSONs.

**Evaluate CUDA profiles (Perlmutter):**

```bash
python -m perf_advisor evaluate bench/cuda/profiles/ \
    --ground-truth bench/ \
    --model claude-opus-4-8 \
    --output eval_results_cuda.json
```

**Evaluate HIP profiles (Frontier):**

```bash
python -m perf_advisor evaluate bench/hip/profiles/ \
    --ground-truth bench/ \
    --model claude-opus-4-8 \
    --output eval_results_hip.json
```

**Rescore without re-running PerfAdvisor** (after editing the rubric or judge prompt):

```bash
python -m perf_advisor evaluate \
    --cached eval_results_cuda.json \
    --ground-truth bench/
```

**Bottleneck-detection-only check** (no judge API calls, fast):

```bash
python -m perf_advisor evaluate bench/cuda/profiles/ \
    --ground-truth bench/ \
    --skip-judge --yes
```

**Per-suggestion detail** (shows judge score and explanation for every suggestion):

```bash
python -m perf_advisor evaluate bench/cuda/profiles/ \
    --ground-truth bench/ \
    --output eval_results_cuda.json \
    --verbose
```

### Scoring

| Axis                 | Method                                                                | Output                      |
| -------------------- | --------------------------------------------------------------------- | --------------------------- |
| Bottleneck detection | Enum gate **AND** scenario-discriminating keyword; deterministic       | detection@1 / @3 / MRR      |
| Suggestion coverage  | LLM judge (default: `claude-opus-4-8`); scores 0 / 1 / 2 per suggestion | raw score + %, or `n/a`     |
| False positives      | Count hypotheses that do not identify the expected bottleneck          | count per run               |

**Detection requires both signals.** A hypothesis counts only when its
`bottleneck_type` is in the scenario's accepted enum set *and* its text names a
mechanism specific to that scenario. The enum alone is insufficient: the advisor's
vocabulary (8 coarse types) is coarser than the benchmark's 8 scenarios, so several
scenarios share a type — all three host-staging MPI scenarios are legitimately
`mpi_latency`. Scoring on the enum alone let a profile-blind model that emits one
hypothesis of each type score a perfect 8/8. `tests/test_eval_scoring.py` holds that
property as a regression test; keep it passing when editing the keyword lists.

The keyword lists are **discriminators, not domain vocabulary**. Terms like `memcpy`,
`h2d`, `kernel launch`, or `p2p` describe the benchmark's subject matter and appear
in nearly any hypothesis about these profiles, so they award credit without evidence
the advisor read the profile. The three `mpi_latency` scenarios have mutually
disjoint keyword sets, and shared fix vocabulary (`cuda-aware`, `host staging`,
`device pointer`) is deliberately in none of them — recommending CUDA-aware MPI for
every MPI profile does not demonstrate the advisor distinguished them.

**Rubric entries must be scenario-specific.** The same discrimination requirement
applies one layer up, to the ground truth itself. An entry that another scenario's
correct answer would satisfy lets an advisor bank coverage points for diagnosing the
*wrong* scenario. An `overlap_missing` entry reading "audit for `cudaDeviceSynchronize`
/ `cudaStreamSynchronize`", for instance, is verbatim the `sync_stall` answer — so on
test_04 an advisor that diagnosed synchronization, missed the transfer/compute pipeline
entirely, and scored a detection MISS could still take 67% coverage. Entries instead
name scenario-specific mechanisms (CUDA IPC for the intra-node exchange, interior/boundary
overlap for the inter-node halo, event-based chunk ordering for the pipeline), and
`tests/test_eval_scoring.py::test_rubric_entries_are_scenario_specific` fails if a new
entry reintroduces the generic-fallback pattern. Genuine exceptions go in that test's allowlist with a
reason, so they stay visible rather than silently accumulating.

**Detection is reported rank-aware.** The advisor emits a *ranked* list, so a correct
diagnosis buried under four wrong ones is not the same result as one it led with.
`detection@1` is the headline; `detection@3` is the practical bar; mean reciprocal
rank (1.00 = always ranked first) combines hit rate and ranking in one number. The
old "any hypothesis anywhere" tally is `detection@∞`, which a model can inflate by
emitting one hypothesis of every kind.

Suggestion coverage reports `n/a`, never `0%`, when nothing was judged — "no
measurement" and "covered nothing" are different results. Skipped/errored judge
calls are excluded from both numerator and denominator; a judge reply that parses as
JSON but carries no valid 0–2 verdict is *skipped*, not scored 0.

**The judge verdict is schema-constrained.** Both provider paths pin the reply to a
JSON schema at decode time (`score` restricted to `0 | 1 | 2`, plus `explanation`),
so a malformed verdict is impossible rather than merely handled. Structured outputs
are model-gated, and `--judge-model` accepts any model, so a judge that rejects the
schema falls back to prompt-only JSON and emits a one-time `RuntimeWarning` naming
the model — the skip-not-zero coercion remains the safety net on that path. If you
see that warning, verdicts for the run are no longer format-guaranteed.

The judge model defaults to `claude-opus-4-8` regardless of the hypothesis model.
Override with `--judge-model`. The suite currently has no optimal-path runs; the
scorer still supports them (`OPTIMAL_PATH_BOTTLENECKS`, empty) and would show them in
the results table while excluding them from detection accuracy and coverage tallies.

### Why the judge is an Opus-tier model

The judge's hard call is not "did the advisor say something relevant" — it is whether
two differently-phrased GPU optimisations are *the same mechanism*. Auditing every
full-credit award Haiku gave on a saved run, most were right, but the wrong ones
failed the same way: crediting a related-but-different fix. It scored "overlap
transfers across streams and batch them" as full coverage of "allocate output buffers
on the device; only transfer final results back" — but overlapping a transfer *hides*
the round-trip, whereas the expected fix *eliminates* it. Those are different
optimisations with different ceilings. Judging that distinction is domain reasoning,
and it is where a stronger model earns its place.

Cost is not a reason to go cheaper. A full 8-scenario eval issues 24 judge calls of
roughly 41K input and 1.4K output tokens in total — about $0.05 on Haiku versus $0.24
on Opus 4.8. Either way the judge is 1–2% of the eval's cost; the hypothesis side
(eight full agentic runs with tool loops) dominates by orders of magnitude. If you are
iterating on the rubric or the scorer and want the fast path, `--judge-model
claude-haiku-4-5` is still there — just re-run on the default before quoting numbers.

### Judge neutrality — a caveat on any cross-model comparison

LLM judges are known to score outputs from their own model family more favourably
than a human labeller would, usually attributed to stylistic affinity rather than
recognising authorship. Since this benchmark ranks Claude, GPT, and Gemini advisors
using a Claude judge, that bias would flatter part of the field.

Three things limit the exposure here: the judge never sees which model produced the
hypotheses (only the hypothesis text is passed to it), scoring is absolute against a
rubric rather than a pairwise "which is better" preference, and each expected
suggestion names a concrete API, which anchors the judgement in something checkable.
All three put this setup in the regime where the effect is documented to be weakest.

Checking for it on an earlier eval run — comparing each advisor's judge-scored
coverage against its detection rate, which is computed by deterministic matching with
no LLM involved — showed no family signature: the GPT advisor sat between the two
Claude models on that ratio, and the self-judged Haiku run had the *lower* of the two
Claude ratios, the opposite of what self-preference predicts. With 6–8 runs per model
and no repeats that was weak evidence of absence rather than proof — worth re-checking
on the next eval.

The larger and less examined exposure is upstream of the judge: `ground_truth_meta.json`
— the text defining the right answer for every scenario — was itself LLM-authored. If
it phrases GPU advice the way one model family phrases GPU advice, advisors from that
family gain an edge before the judge is ever called, and no judge swap touches it. The
ratio check above cannot see this, because rubric-level bias inflates detection and
coverage together and leaves the ratio flat. Treat cross-family gaps of a few points
as inconclusive, and prefer within-family comparisons when you can.

The way to settle either question is to hand-label a subset of judgements and measure
the judge against your own labels. That is the only step here that measures bias
rather than working around it, and it would quantify the leniency issue at the same
time.

**Known-true secondary observations (`also_true`).** A profile can legitimately
contain more than the injected deficiency — some of it an artefact of the harness
itself. Counting a correct report of one of those as a false positive conflates two
opposite behaviours: inventing an irrelevant problem, and spotting a second real one.
Scenarios may therefore declare an `also_true` list in `ground_truth_meta.json`:

```json
"also_true": [
  { "bottleneck_type": ["synchronization", "mpi_imbalance"],
    "keywords": ["mpi_barrier", "barrier"],
    "why": "harness barrier_sync() runs inside the profiled region once per pass" }
]
```

Matching uses the same enum-**AND**-keyword conjunction as detection, and the
`bottleneck_type` values must be disjoint from the scenario's own accept set, so an
entry cannot become a blanket excuse. **Detection is unaffected** — an `also_true`
match never counts as finding the injected bottleneck; it only removes the hypothesis
from the false-positive tally, and the count of exclusions is shown next to it
(`4 (+2 also-true)`) so the adjustment stays auditable. Every entry carries a `why`
recording the evidence, and entries should be deleted as the underlying benchmark
defects are fixed.

**False positives are reported as a rate.** The raw count is not comparable across
models — one emitting 4 hypotheses and one emitting 10 are not on the same scale, and
a model that errors out early scores a flatteringly low count. The table shows both
(`3/4 (75%)`); the rate is what belongs in a cross-model comparison.

**The summary prints a profile-blind floor.** Every run reports what a constant
advisor that never opens a profile would score on the same scenarios. If a real
model is not clearly above that line, the numbers are not measuring analysis — this
is the property the whole eval rests on, made visible on every run rather than only
when someone runs the test suite.

**`--repeats N` for error bars.** The agent loop is stochastic, so one sample per
(model, scenario) cannot separate a real difference between models from run-to-run
noise. With `N>1`, detection@1, detection@3 and MRR are reported as **mean ± sample
SD across repetitions**, with the per-repeat counts shown:

```
Detection@1:        4.2 / 8  (52% ± 6)  [5 repeats: 4, 4, 5, 4, 4]
```

One observation per repetition is the unit, not one per (scenario × repetition):
repeats of the same scenario are correlated and scenarios differ in difficulty, so
pooling them would understate the spread. `N=5` is about the minimum for an SD to
mean anything. Stage 1 (profile parsing) is repeated along with Stage 2, so
wall-clock scales linearly — `--workers` offsets it.

A **per-scenario stability table** is also printed, and is usually the more useful
output: the aggregate cannot distinguish *found it every time* from *found it
sometimes* from *never found it*, three different capability claims that average to
the same number. Repeats that errored are excluded from the denominator — a crashed
run is missing data, not a miss.

**What the error bars do and don't cover.** Detection, MRR and the false-positive
rate are computed by deterministic matching with no LLM involved, so repeating the
advisor gives a complete interval for them. Suggestion coverage additionally passes
through the LLM judge, so its spread is *combined* advisor + judge variance — which
is the right thing to publish if the interval is meant to answer "what would a
replication of this pipeline produce". What no amount of sampling fixes is judge
*bias*: error bars measure precision, not accuracy. A systematically lenient judge
yields a tight interval around a wrong mean. That needs the hand-labelled
calibration subset, not more repeats.

**Repeats are not a substitute for more scenarios.** Repetitions resolve *"is model B
better than A on these 8 scenarios"*. They say little about *"is B better at GPU
profile analysis"* — with n=8, a single scenario is 12.5 points, and that limit is
set by the scenario count, not the repeat count.

**Token accounting.** Input, output and cache-read tokens are recorded per run and
totalled in the summary. Cost and tokens are first-order for a cross-LLM comparison,
and `elapsed_s` alone conflates API latency with quality. These cannot be recovered
after the fact without paying for another eval pass, so they are captured on every
run rather than opt-in. The judge runs at the provider's default sampling
temperature and is sampled once per suggestion, so within a single run coverage
differences of a few points are within noise — use `--repeats` for a variance estimate.

## Limitations

- **The eval measures recall, not precision.** Every captured profile contains an
  injected deficiency, so nothing here checks whether an advisor *over-reports* — on
  these profiles the false-positive count is adjudicated through `also_true` and is a
  negotiated number, not a clean measurement, and there is no bottleneck-free control
  profile. The machinery to add one back is retained but unused: `OPTIMAL_PATH_BOTTLENECKS`
  in `eval/scorer.py` (empty) and `run_halo_exchange_impl(..., device_ptrs=true)` (no
  caller); see `todo_list.md`.
- **HIP profiles include setup and warmup.** Capture is scoped to the timed loop on
  CUDA (`cudaProfilerStart/Stop`), but on HIP `roctxProfilerPause` returns success
  without actually gating rocprof-sys 7.2.0's trace, so warmup passes are still
  recorded (HIP test_01 captures 150k dispatches against CUDA's 100k). Model-vs-model
  comparison within one platform is unaffected, but CUDA-vs-HIP *absolute* numbers are
  not apples-to-apples.
