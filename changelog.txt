2026-04-07
----------
- Fix preflight token overestimate for OpenAI/Gemini: `estimate_json_tokens` now accepts
  a `provider` parameter with calibrated chars/token ratios (anthropic: 2.5, openai/gemini: 3.5);
  `cmd_analyze` and `cmd_compare` pass `resolved_provider` through
- Show input token estimate as a range (5 turns → max_turns) for OpenAI/Gemini rather than
  a single worst-case figure; range collapses to a single number when max_turns ≤ 5
- Add per-turn context size display to OpenAI and Gemini backends (matching Anthropic):
  OpenAI uses `response.usage.prompt_tokens` and `prompt_tokens_details.cached_tokens`;
  Gemini uses `usage_metadata.prompt_token_count` and `cached_content_token_count`
- Fix OpenAI backend multi-turn bug: when `finish_reason == "stop"` but the model outputs
  prose instead of a JSON array, the loop now injects a recovery prompt and continues rather
  than returning `[]` immediately after a single turn
- Fix OpenAI backend file saving: prompt and response are now saved unconditionally on stop,
  not only when hypotheses are successfully extracted
- Remove rate-limiting feature (`--rate-limit-tpm`, `_rate_limit_sleep`, `_token_window`,
  `rate_limit_tpm` param from `_run_api` and `run_agent`): Anthropic's rate limit does
  not apply to cache reads; with sliding cache the non-cached increment per turn is only
  ~1–3k tokens, making the rate limit effectively unreachable in normal use
- Update per-turn context display in `_run_api`: shows `Context size ≈ X,XXX tokens
  (Y,YYY cached)` from turn 2 onwards, where cached ≈ previous turn's total context;
  turn 1 shows no cached count since nothing is in cache yet
- Add `--rate-limit-tpm` flag to `analyze`: when set, `_run_api` tracks a 60-second
  rolling token window (deque of monotonic timestamp + exact context size from
  `response.usage`) and calls `_rate_limit_sleep` before each turn; `_rate_limit_sleep`
  loops until the window has room, sleeping the exact number of seconds until the oldest
  entry expires and printing a notification to the user each time it sleeps; single
  requests that alone exceed the limit are passed through without sleeping; no-ops for
  non-Anthropic providers; `run_agent` gains `rate_limit_tpm` parameter
- Add per-turn context size display to `_run_api`: prints `Context size ≈ X,XXX tokens`
  before each API call (turn 1 uses heuristic estimate from messages + schemas + system
  prompt; subsequent turns use exact prior context + output + tool result estimate)
- Move turn header display to before the API call in `_run_api` so it appears with the
  context size line rather than after the (potentially long) API wait
- Fix `cache_control` 4-block limit error and restore true sliding cache in `_run_api`:
  keep the 2 most-recent per-turn user messages marked (`_cache_prev` + `_cache_pprev`),
  stripping only the oldest on each advance; total markers stay at 4 (system + preseed +
  2 floating), which is Anthropic's maximum; this gives true sliding cache where turn N
  reads everything through turn N-1 from cache (0.10×) rather than only the pre-seeded
  content
- Add Anthropic sliding prompt cache to `_run_api` in `loop.py`: system prompt and
  pre-seeded profile summary marked with `cache_control: ephemeral` on turn 1 (cache
  write at 1.25×); each subsequent turn marks the last tool-result block to extend the
  cached prefix (cache read at 0.10× for all prior content, cache write for the new
  increment); result is ~75–80% reduction in billable input tokens on a typical multi-turn
  run; `cache_creation_tokens` and `cache_read_tokens` tracked per turn and summed
- Return `_run_api`, `_run_openai`, `_run_gemini` as 5-tuples
  `(hypotheses, input, cache_creation, cache_read, output)`; `run_agent` populates
  `token_usage` with `cache_creation_tokens` and `cache_read_tokens`; OpenAI and
  Gemini return 0 for both cache fields
- Add `estimate_total_session_tokens` and `estimate_cache_breakdown` to `preflight.py`:
  `estimate_total_session_tokens` computes total input across N stateless turns
  (N×initial + increment×N(N-1)/2); `estimate_cache_breakdown` computes expected
  cache_creation, cache_read, and cost-equivalent for Anthropic sliding cache
- Update `cmd_analyze` preflight display: Anthropic shows cache_write / cache_read /
  cost-equiv breakdown; other providers show total session estimate with a note that
  OpenAI applies automatic ~50% caching
- Update `cmd_analyze` post-run token display: when cache fields are non-zero, show
  non-cached / cache-write / cache-read / out separately with total and cost-equivalent;
  fall back to existing `in / out` display for non-caching providers
- Update README: add "Prompt caching" subsection explaining per-provider caching
  behavior; update "Pre-flight estimate" with new display format examples; add note to
  "Minimizing token expenditure" that Anthropic caching is automatic
- Update README: document `--max-turns` flag, wrap-up warning, and forced final
  turn behavior in Useful Flags and Minimizing Token Expenditure sections;
  add note to Pre-flight Estimate section that output range reflects `--max-turns`
- Add turn-limit safety net to agent loop (`loop.py`): instead of raising
  `RuntimeError` when `max_turns` is exhausted, inject a wrap-up warning
  `WARN_TURNS_BEFORE_LIMIT` (3) turns before the limit (bundled with tool
  results so no extra API call); on the final turn, bundle the forced-output
  prompt with tool results; after the loop, make one additional API call
  without tools to extract whatever hypotheses the model has accumulated;
  applies to all three backends (Anthropic, OpenAI, Gemini); Gemini uses
  `Part.from_text` for the warning and a standalone `chat.send_message` for
  the forced turn; returns `[]` (with a warning log) rather than raising if
  no hypotheses are produced even after the forced turn
- Add `--max-turns` flag to `analyze` subcommand: overrides `MAX_TURNS=20`
  default; wired through to `run_agent`; preflight output/range display
  uses the resolved value rather than the hardcoded constant
- Improve pre-flight token estimate accuracy: replace single char/4 heuristic
  with content-aware `estimate_prose_tokens` (4 chars/token) and
  `estimate_json_tokens` (2.5 chars/token) in `preflight.py`; `analyze` applies
  prose ratio to the system prompt and JSON ratio to summary + tool schemas;
  `compare` applies prose ratio to the system prompt and JSON ratio to the
  compare prompt (which is ~95% JSON); reduces underestimate from ~34% to ~5%
  on a real profile comparison run (13k estimated → 21k vs 19,771 actual)
- Add pre-flight token estimate to `analyze` and `compare` subcommands: after
  `compute_profile_summary()` (analyze) or `compute_profile_diff()` (compare) and
  before the LLM call, print input/output token estimates and prompt for confirmation;
  confirmation is skipped when `--yes`, `--quiet`, `--json`, or stdin is non-interactive;
  estimate is suppressed only by `--quiet` or `--json`
- Add `--yes` flag to `analyze` and `compare`: skip confirmation prompt
- Add `--exact-token-count` flag to `analyze` and `compare`: use Anthropic's
  `client.messages.count_tokens()` API for an exact input count instead of the char/4
  heuristic; no-ops with a note for non-Anthropic providers
- Add `nsight_agent/agent/preflight.py`: `estimate_tokens_heuristic(*texts)` and
  `count_tokens_exact(provider, model, system_prompt, *user_texts, tools=None)`
- Consolidate `--provider` flag into `--model`: `_parse_provider_and_model` now
  accepts only the model string; a bare provider name (e.g. `--model=openai`)
  is treated as a provider-only shorthand that selects the default model for
  that provider; the explicit `--provider` flag is removed from both `analyze`
  and `compare` subparsers; `run_agent` and `run_compare` signatures updated
  accordingly; README Provider Selection section updated with new examples

2026-04-07
----------
- Add `compare` subcommand: single-shot LLM comparison of two `.sqlite` profiles
  producing a structured `ComparisonReport` (narrative + key differences table
  ordered by magnitude); no tool-use loop — both summaries and a pre-computed
  structural diff are injected into one prompt
- Add `nsight_agent/analysis/diff.py`: `compute_profile_diff()` aligns two
  `ProfileSummary` objects and determines comparison mode:
    - `phase_aware` — same phase count + names; full per-phase analysis
    - `summary` — phases differ but Jaccard kernel overlap >= 20%; per-kernel diff included
    - `summary_no_kernel` — phases differ and overlap < 20%; top-level metrics only
  Jaccard similarity used for kernel overlap; status message shown before LLM call
  (e.g. "Phases do not match (A has 4, B has 6) and kernel overlap is 8%...")
- Add `nsight_agent/agent/compare.py`: `run_compare()` single-shot backends for
  Anthropic, OpenAI, Gemini, and claude_code; saves `{stem_a}_vs_{stem_b}_{ts}_prompt.txt`
  and `_response.txt` alongside the profile
- Add `ScalarDiff`, `KernelDiff`, `MemcpyDiff`, `MpiDiff`, `ProfileDiff`,
  `ComparisonDiff`, `ComparisonReport` Pydantic models to `analysis/models.py`
- Add `_print_phase_table()` helper to `__main__.py`; `compare` always prints both
  phase tables before the LLM call; key differences table color-codes by magnitude
- Output saved to `{stem_a}_vs_{stem_b}_{ts}_output.txt` next to profile A

- Save terminal output to `{stem}_{ts}_output.txt` alongside prompt/response files
  after each `analyze` run: `Console(record=True)` captures all Rich table output;
  a `log` callable is threaded through `run_agent` and all four backends so the
  turn-by-turn agent output (previously raw `print()`) is routed through the same
  console and captured; output file is not written in `--json` mode

- Remove `_ensure_indexes` and the writable connection in `NsysProfile`: the tool
  previously opened a writable connection on first use to add query-accelerating
  indexes to the profile file; SQLite does not support temporary indexes on
  non-temporary tables, so there is no non-modifying alternative; the ~25% local
  analysis speedup was not worth the hidden file modification given that runtime
  is dominated by LLM round-trips
- Add `has_table` guards to `compute_gpu_sync_time` and `compute_gpu_memcpy_time`:
  previously both would crash with `OperationalError` on profiles that lack
  `CUPTI_ACTIVITY_KIND_SYNCHRONIZATION` or `CUPTI_ACTIVITY_KIND_MEMCPY`
- Add docstring note to `PhaseSummary` clarifying that `total_gpu_idle_s` counts
  only inter-kernel gaps, not leading/trailing idle at phase boundaries
- Add "Provenance" section to README noting the tool was vibe coded and that
  metrics should be treated as starting points rather than ground truth

2026-04-06
----------
- Add phase-scoped time window support to agent tools: `top_kernels`, `gap_histogram`,
  `memcpy_summary`, `mpi_summary`, `nvtx_ranges`, and `stream_summary` now accept optional
  `start_ns`/`end_ns` parameters; when provided they query only within that window using
  existing internal `_window_*` helpers; tool schemas updated with parameter descriptions
  that direct the agent to use phase timestamps from the pre-seeded context
- Add `_window_memcpy_by_kind`, `_window_streams`, `_window_nvtx_ranges` to `metrics.py`
  (windowed counterparts for the three functions that previously lacked them)
- Add `start_ns`/`end_ns` fields to `PhaseSummary` (absolute CUPTI nanosecond timestamps);
  the pre-seeded phase summary previously only had `start_s`/`end_s` (seconds), which the
  agent could not use directly as tool window arguments
- Add `--allow-app-knowledge` flag to `analyze`: by default the system prompt
  now constrains the model to ground all suggestions in profile data only;
  `--allow-app-knowledge` restores the previous behavior where the model may
  draw on training knowledge about the application (e.g. suggest named env vars
  inferred from kernel names)
- Full kernel name resolution (todo item 2): switch GROUP BY from shortName to
  demangledName so each template instantiation is a distinct row; add
  `_normalize_demangled()` to strip SFINAE prefix and `(T2)` suffix; add
  `short_name` field to `KernelSummary` for CLI display; fall back to shortName
  on older profiles that lack the demangledName column
- Add `action_category` field to hypothesis schema (todo item 5): enum of
  `runtime_config` / `launch_config` / `code_optimization` / `algorithm`,
  ordered by effort; boundary rule (what vs. how) included in system prompt
- Add "Action" column (magenta, abbreviated) to hypotheses table in `cmd_analyze`
- Update `README.md`: document `action_category` field in hypothesis output
- Add `eval/score.py` — standalone independent hypothesis evaluator; scores
  nsight-agent hypothesis JSON using gpt-5.4 on six criteria (bottleneck
  alignment, addressability, mechanistic plausibility, specificity/actionability,
  testability, generality) plus implementation-cost and correctness-risk flags
  and a list-level redundancy score; writes `.eval.json` alongside input
- Add `eval/context_example.yaml` — template for injecting application/phase
  context into the evaluator prompt (critical for addressability scoring)
- Add `eval/` to `.gitignore` — evaluator is intentionally separate from nsight-agent
- Add rich terminal table to `eval/score.py` — displays per-hypothesis scores (color-coded
  1–5), implementation-cost and correctness-risk flags, and per-hypothesis average; footer
  shows redundancy score and overall aggregate
- Update `eval/score.py` table and output files: rename column "Hypothesis" → "Suggestion"
  (shows verbatim suggestion text truncated to 50 chars); save full LLM prompt to
  *.eval.prompt.txt; save plain-text table + rationales to *.eval.table.txt
- Add verdict + knowledge_source to `eval/score.py` — LLM now gives an explicit
  strongly_recommended/recommended/uncertain/not_recommended verdict per hypothesis with
  a 2-3 sentence rationale; knowledge_source field (context/model/both) flags when the
  verdict draws on the model's own training knowledge of the application vs. injected context;
  verdict column added to the rich table with rationales printed below; prompt strengthened
  to be opinionated and treat injected context as authoritative
