  not only when hypotheses are successfully extracted
- Remove rate-limiting feature (`--rate-limit-tpm`, `_rate_limit_sleep`, `_token_window`,
  `rate_limit_tpm` param from `_run_api` and `run_agent`): Anthropic's rate limit does
  not apply to cache reads; with sliding cache the non-cached increment per turn is only
  ~1–3k tokens, making the rate limit effectively unreachable in normal use
- Update per-turn context display in `_run_api`: shows `Context size ≈ X,XXX tokens
  (Y,YYY cached)` from turn 2 onwards, where cached ≈ previous turn's total context;

- Add `--rate-limit-tpm` flag to `analyze`: when set, `_run_api` tracks a 60-second
  rolling token window (deque of monotonic timestamp + exact context size from
  `response.usage`) and calls `_rate_limit_sleep` before each turn; `_rate_limit_sleep`
  loops until the window has room, sleeping the exact number of seconds until the oldest
  entry expires and printing a notification to the user each time it sleeps; single
  requests that alone exceed the limit are passed through without sleeping; no-ops for
  non-Anthropic providers; `run_agent` gains `rate_limit_tpm` parameter
- Add per-turn context size display to `_run_api`: prints `Context size ≈ X,XXX tokens`
  before each API call (turn 1 uses heuristic estimate from messages + schemas + system
  prompt; subsequent turns use exact prior context + output + tool result estimate)
- Move turn header display to before the API call in `_run_api` so it appears with the
  context size line rather than after the (potentially long) API wait
- Fix `cache_control` 4-block limit error and restore true sliding cache in `_run_api`:
  keep the 2 most-recent per-turn user messages marked (`_cache_prev` + `_cache_pprev`),
  stripping only the oldest on each advance; total markers stay at 4 (system + preseed +
  2 floating), which is Anthropic's maximum; this gives true sliding cache where turn N
  reads everything through turn N-1 from cache (0.10×) rather than only the pre-seeded
  content
- Add Anthropic sliding prompt cache to `_run_api` in `loop.py`: system prompt and
  pre-seeded profile summary marked with `cache_control: ephemeral` on turn 1 (cache
  write at 1.25×); each subsequent turn marks the last tool-result block to extend the
  cached prefix (cache read at 0.10× for all prior content, cache write for the new
  increment); result is ~75–80% reduction in billable input tokens on a typical multi-turn
  run; `cache_creation_tokens` and `cache_read_tokens` tracked per turn and summed
- Return `_run_api`, `_run_openai`, `_run_gemini` as 5-tuples
  `(hypotheses, input, cache_creation, cache_read, output)`; `run_agent` populates
  `token_usage` with `cache_creation_tokens` and `cache_read_tokens`; OpenAI and

- Add `estimate_total_session_tokens` and `estimate_cache_breakdown` to `preflight.py`:
  `estimate_total_session_tokens` computes total input across N stateless turns
  (N×initial + increment×N(N-1)/2); `estimate_cache_breakdown` computes expected
  cache_creation, cache_read, and cost-equivalent for Anthropic sliding cache
- Update `cmd_analyze` preflight display: Anthropic shows cache_write / cache_read /

- Update `cmd_analyze` post-run token display: when cache fields are non-zero, show
  non-cached / cache-write / cache-read / out separately with total and cost-equivalent;

- Update README: add "Prompt caching" subsection explaining per-provider caching

- Update README: document `--max-turns` flag, wrap-up warning, and forced final
  turn behavior in Useful Flags and Minimizing Token Expenditure sections;
  add note to Pre-flight Estimate section that output range reflects `--max-turns`
- Add turn-limit safety net to agent loop (`loop.py`): instead of raising
  `RuntimeError` when `max_turns` is exhausted, inject a wrap-up warning
  `WARN_TURNS_BEFORE_LIMIT` (3) turns before the limit (bundled with tool
  results so no extra API call); on the final turn, bundle the forced-output

- Add `--max-turns` flag to `analyze` subcommand: overrides `MAX_TURNS=20`

- Improve pre-flight token estimate accuracy: replace single char/4 heuristic
  with content-aware `estimate_prose_tokens` (4 chars/token) and
  `estimate_json_tokens` (2.5 chars/token) in `preflight.py`; `analyze` applies
  prose ratio to the system prompt and JSON ratio to summary + tool schemas;

2026-04-07

- Add `compare` subcommand: single-shot LLM comparison of two `.sqlite` profiles
  producing a structured `ComparisonReport` (narrative + key differences table
  ordered by magnitude); no tool-use loop — both summaries and a pre-computed
  structural diff are injected into one prompt
- Add `perf_advisor/analysis/diff.py`: `compute_profile_diff()` aligns two
  `ProfileSummary` objects and determines comparison mode:
    - `phase_aware` — same phase count + names; full per-phase analysis
    - `summary` — phases differ but Jaccard kernel overlap >= 20%; per-kernel diff included
    - `summary_no_kernel` — phases differ and overlap < 20%; top-level metrics only
  Jaccard similarity used for kernel overlap; status message shown before LLM call
  (e.g. "Phases do not match (A has 4, B has 6) and kernel overlap is 8%...")
- Add `perf_advisor/agent/compare.py`: `run_compare()` single-shot backends for
  Anthropic, OpenAI, Gemini, and claude_code; saves `{stem_a}_vs_{stem_b}_{ts}_prompt.txt`
  and `_response.txt` alongside the profile
- Add `ScalarDiff`, `KernelDiff`, `MemcpyDiff`, `MpiDiff`, `ProfileDiff`,
  `ComparisonDiff`, `ComparisonReport` Pydantic models to `analysis/models.py`
- Add `_print_phase_table()` helper to `__main__.py`; `compare` always prints both
  phase tables before the LLM call; key differences table color-codes by magnitude
- Output saved to `{stem_a}_vs_{stem_b}_{ts}_output.txt` next to profile A

- Save terminal output to `{stem}_{ts}_output.txt` alongside prompt/response files
  after each `analyze` run: `Console(record=True)` captures all Rich table output;
  a `log` callable is threaded through `run_agent` and all four backends so the
  turn-by-turn agent output (previously raw `print()`) is routed through the same
  console and captured; output file is not written in `--json` mode

- Remove `_ensure_indexes` and the writable connection in `NsysProfile`: the tool
  previously opened a writable connection on first use to add query-accelerating
  indexes to the profile file; SQLite does not support temporary indexes on
  non-temporary tables, so there is no non-modifying alternative; the ~25% local
  analysis speedup was not worth the hidden file modification given that runtime
  is dominated by LLM round-trips
- Add `has_table` guards to `compute_gpu_sync_time` and `compute_gpu_memcpy_time`:
  previously both would crash with `OperationalError` on profiles that lack
  `CUPTI_ACTIVITY_KIND_SYNCHRONIZATION` or `CUPTI_ACTIVITY_KIND_MEMCPY`
- Add docstring note to `PhaseSummary` clarifying that `total_gpu_idle_s` counts
  only inter-kernel gaps, not leading/trailing idle at phase boundaries
- Add "Provenance" section to README noting the tool was vibe coded and that
  metrics should be treated as starting points rather than ground truth
