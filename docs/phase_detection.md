# Phase Detection

## Purpose and Motivation

PerfAdvisor segments a profile into a flat, ordered list of non-overlapping **phases** before running any analysis. The goal is to make per-phase analysis meaningful: each phase is a time window over which the performance characteristics are sufficiently homogeneous that a single set of bottlenecks and optimization strategies applies to the whole window. If two adjacent windows have the same kernel mix and the same GPU utilization profile, they have the same bottleneck by definition and should be merged. If they differ, they may have different bottlenecks that require separate attention.

This matters practically because many profiling metrics — GPU utilization, dominant kernel, memory bandwidth, MPI imbalance — vary widely across different parts of a run. A training loop that alternates between a data-loading phase and a compute phase will have a very different optimization story in each half. Averaging those metrics across the entire profile obscures both bottlenecks. Phase-aware analysis surfaces each cleanly.

The user controls granularity through `--max-phases`. A large value produces fine-grained phases (e.g., individual solver passes); a small value forces merging until only the broadest structural divisions remain (e.g., "initialization", "compute", "finalize"). `--max-phases 1` skips segmentation entirely and returns a single phase covering the full profile span.

## Challenges

**No universal definition of "phase"**

A phase is not formally defined by any profiler or GPU runtime. The concept is application-level: what constitutes a phase in a lattice QCD CG solver is not the same as in a deep learning training loop or a molecular dynamics simulation. Any general-purpose algorithm must infer phase structure from low-level hardware events (kernel launches, memory transfers, synchronization) without knowing anything about the application's intent.

**Wide dynamic range in phase lengths**

Initialization and finalization may last milliseconds while the main compute phase runs for hours. An algorithm that works well at one scale may collapse short phases into neighbors or split long homogeneous phases unnecessarily.

**The boundary placement problem**

The quality of the final segmentation depends critically on where initial candidate boundaries are placed. If two genuinely different phases never receive a boundary candidate between them, no amount of subsequent merging or labeling can recover the distinction. Conversely, spurious boundary candidates generate spurious segments that must be merged away, which can introduce errors if the merge criterion is coarse.

**Phase transitions without idle gaps**

A well-optimized pipelined workload may have no GPU idle time at all, even across what are semantically distinct phases. Gap-based boundary detection produces nothing in this case. The algorithm must be able to detect phase transitions from changes in kernel activity patterns alone.

**Labeling without annotations**

Human-readable phase names require either application-supplied NVTX annotations or heuristic inference from kernel names. NVTX annotations are absent in many production HPC codes, and kernel names from vendor libraries (cuBLAS, cuDNN, NCCL) are often mangled, autogenerated, or generic — everything is a GEMM or a collective. The labeling stage must degrade gracefully in both cases.

## Pathologies a Robust Algorithm Must Handle

**Profiles captured mid-run**
Nsight Systems is frequently started after warmup or targeted at a specific region. The "initialization" phase may be entirely absent. Phase boundaries at the profile edges may be artificial cut points rather than genuine application transitions.

**Warmup and JIT compilation iterations**
ML frameworks (PyTorch, JAX) and CUDA JIT compilers spend the first N iterations doing graph capture, kernel compilation, or CUDA context setup. These iterations resemble the steady-state compute loop in structure but have very different timing and are not representative of steady-state performance.

**Purely CPU-bound phases**
Checkpoint loading, data preprocessing, MPI-only communication, and some initialization steps have no GPU kernel activity at all. They appear as GPU-idle windows but are meaningful phases, not noise. An algorithm that only reasons about GPU events cannot distinguish a meaningful CPU-only phase from an uninstrumented gap.

**Pipelined multi-stream workloads**
When compute, memory copy, and communication are overlapped across multiple CUDA streams, there may be no GPU idle time across the entire profile. Gap-based boundary candidates never fire. Only a change in the *distribution* of kernel activity across streams signals a phase transition.

**Thousands of short identical iterations**
A CG solver or a training loop may execute the same small kernel sequence ten thousand times in succession. These iterations are not separate phases — they share the same bottleneck and should be merged into a single phase. The algorithm must collapse them without losing the ability to distinguish a CG solve phase from a gauge-fixing phase that happens to use some of the same kernels.

**Multiple distinct solve blocks**
Some applications perform several CG solves with different configurations (e.g., plain CG followed by CG with eigenvector deflation). These solve blocks are semantically distinct phases even though each internally looks like thousands of identical iterations. The algorithm must merge within each block but preserve the boundary between blocks.

**Vendor library kernels with uninformative names**
Kernels from cuBLAS, cuDNN, cuSPARSE, and NCCL have mangled or autogenerated names that do not reflect application intent. A profile dominated by vendor library calls may have identical dominant-kernel labels across phases that are structurally very different (e.g., a convolution forward pass vs. a gradient computation, both labeled `volta_sgemm_...`).

**MPI load imbalance masquerading as a phase**
A rank that finishes its compute work early and spins in `MPI_Waitall` looks like it has a dedicated "wait" phase. This is an artifact of load imbalance, not a genuine application phase. Per-rank phase detection must handle this, and cross-rank phase alignment must account for it.

**Malformed or misleading marker annotations**
NVTX or rocTX ranges may be left open (no matching end event), have empty or null text, overlap in ways that violate the expected nesting structure, or use names that are unhelpful (e.g., `"step"` repeated 10,000 times). The algorithm must be robust to all of these without crashing or producing nonsensical output.

**Extremely short profiles**
A profile capturing a single kernel launch or a 200ms window. Phase detection should produce one phase gracefully rather than emitting a degenerate result or crashing.

**Extremely long profiles**
A multi-hour training run produces a SQLite file measured in gigabytes with millions of kernel rows. Any algorithm that loads all kernel timestamps into memory or executes O(n²) queries against the full kernel table will time out or OOM. All heavy computation must be done in aggregated SQL queries.

## Algorithm

`detect_phases()` (in `analysis/phases.py`) returns a flat, ordered, non-overlapping list of
`PhaseWindow` objects. The algorithm uses **kernel-distribution change-point detection**: the
timeline is divided into fixed-width bins, each bin is fingerprinted as a normalized probability
distribution over kernel and memory-copy activity, and boundaries are placed where adjacent
distributions diverge most. NVTX and MPI signals are retained as supplementary candidates. The
merge criterion uses Jensen-Shannon divergence on the full kernel distribution; final segmentation
is globally optimal via dynamic programming.

Pass `--max-phases 1` to skip segmentation entirely and return a single phase covering the full
profile span.

---

### Step 1 — Compute Profile Bounds

Query `MIN(start)` and `MAX(end)` across all event tables: CUDA kernels, memory copies, CUDA
runtime API calls, NVTX events, OS runtime calls, and MPI event tables. This gives `t0` and `t1`
— the full span of recorded activity. If both are zero (empty profile), return a single
`PhaseWindow("unknown", 0, 0)` immediately.

If `max_phases <= 1`, skip all subsequent steps and return a single `PhaseWindow("profile", t0, t1)`.

---

### Step 2 — Build the Kernel Vocabulary

Query `CUPTI_ACTIVITY_KIND_KERNEL` for total GPU time per kernel across the entire profile,
ordered by total time descending. Take the top `_TOP_K` names. This ordered list is the
**vocabulary** — all feature vectors computed later are defined over the same set of vocabulary
keys, plus two reserved pseudo-kernel entries:

- `"__memcpy__"` — represents all memory copy activity (H2D, D2H, D2D)
- `"__idle__"` — represents bins within a segment that had no GPU or memcpy activity (encodes GPU utilization implicitly; see Step 7)
- `"__other__"` — represents GPU kernel time from kernels outside the top-`_TOP_K`

**Kernel names use demangled names, not short names.** Short names (e.g., `Kernel3D`,
`Reduction2D`) are too coarse for vocabulary purposes — in template-heavy codebases like QUDA,
a single short name covers dozens of physically distinct operations that have different
performance characteristics and different optimization strategies. Demangled names (e.g.,
`quda::Kernel3D<dslash_functor<...>>`) correctly distinguish these instantiations and produce a
vocabulary that reflects actual computational character rather than launch-wrapper identity.

The same fallback pattern used in `metrics.py` applies here: check whether the `demangledName`
column exists in `CUPTI_ACTIVITY_KIND_KERNEL` (it is absent in some older Nsight Systems
versions). If present, join `StringIds` twice — once on `shortName` and once on `demangledName`
— and use `COALESCE(demangled_value, short_value)` so that rows where the demangled name is null
still resolve to the short name. After resolving, apply `_normalize_demangled()` (already
implemented in `metrics.py`) to strip SFINAE return-type prefixes and `(T2)` artifacts injected
by nvcc. If the `demangledName` column is absent entirely, fall back to short names throughout.

The vocabulary is computed once per `detect_phases()` call and threaded through all subsequent
steps. If `CUPTI_ACTIVITY_KIND_KERNEL` is absent (profile has no GPU kernels), the vocabulary is
empty and the algorithm degrades gracefully: all bins will produce idle feature vectors and phases
will be labeled by NVTX coverage or as `"idle"`.

---

### Step 3 — Bin the Timeline

Divide `[t0, t1]` into fixed-width bins of duration:

```
bin_width_ns = clamp(
    (t1 - t0) / _N_BINS,
    _BIN_WIDTH_FLOOR_NS,
    _BIN_WIDTH_CEIL_NS
)
```

`_N_BINS` is a fixed constant (not a function of `max_phases`). This is deliberate: bin width
governs **boundary precision**, and precision should not degrade just because the user asks for
fewer phases. A user who sets `max_phases=2` still expects the single detected boundary to be
accurately placed — not coarsened by a factor of three relative to `max_phases=6`. The influence
of `max_phases` on how many boundary candidates are generated is handled separately in Step 5a,
after binning.

The number of bins is `n_bins = ceil((t1 - t0) / bin_width_ns)`. Bin `i` covers the interval
`[t0 + i × bin_width_ns,  t0 + (i+1) × bin_width_ns)`.

**Kernel aggregation** uses center-based attribution: a kernel whose midpoint
`(start + end) / 2` falls in bin `i` is attributed entirely to bin `i`. This avoids
double-counting boundary-straddling kernels and requires only a single O(n_kernels) SQL pass.
Kernel names use demangled names with the same fallback pattern as Step 2 —
`COALESCE(sd.value, s.value)` after joining `StringIds` on both `shortName` and `demangledName`,
with `_normalize_demangled()` applied in Python after the query returns. The `GROUP BY` uses
`COALESCE(k.demangledName, k.shortName)` to group by the integer FK before string resolution,
which is more efficient than grouping by the resolved string:

```sql
SELECT
    CAST(((k.start + k.end) / 2 - :t0) / :bin_width AS INTEGER) AS bin_idx,
    COALESCE(sd.value, s.value) AS kernel_name,
    SUM(k.end - k.start) AS kernel_ns
FROM CUPTI_ACTIVITY_KIND_KERNEL k
JOIN StringIds s ON k.shortName = s.id
LEFT JOIN StringIds sd ON k.demangledName = sd.id   -- omitted if demangledName column absent
WHERE k.start >= :t0 AND k.end <= :t1
GROUP BY bin_idx, COALESCE(k.demangledName, k.shortName)
```

`_normalize_demangled()` is applied to each `kernel_name` value in Python after the query
returns, before the results are aggregated into `bins[]`.

**Memcpy aggregation** uses the same structure against `CUPTI_ACTIVITY_KIND_MEMCPY`, with
`"__memcpy__"` substituted for the kernel name. The result is merged into the per-bin data.

The output of this step is a list of raw dicts: `bins[i]` maps kernel name (or `"__memcpy__"`)
to total nanoseconds of activity centered in that bin. Bins with no activity are represented as
empty dicts.

---

### Step 4 — Compute Feature Vectors

The raw bin data from Step 3 is a dict of nanoseconds per kernel name — useful for accounting
but not yet in a form that can be compared across bins. The comparison metric used later (Jensen-
Shannon divergence, Step 5a) operates on **probability distributions**: vectors of non-negative
values that sum to 1.0. This step converts each raw bin dict into such a distribution.

The key insight behind normalization is that we want to compare the **character** of GPU work
across bins, not the volume. Two bins where 80% of GPU time goes to kernel A and 20% to kernel B
represent the same computational pattern regardless of whether the absolute GPU time in those bins
is 10ms or 1000ms. Normalizing to shares captures the mix; the absolute activity level is
separately visible in `gpu_util`. Two bins with the same kernel mix but different absolute activity
should be treated as similar (same bottleneck structure) in the merge criterion — and with a
normalized representation, they will be.

For each bin, the normalization proceeds as follows:

1. **Sum all raw times**: `total_ns = sum(bins[i].values())`, combining all kernel and memcpy
   entries.

2. **Handle idle bins specially**: if `total_ns == 0`, the bin has no GPU or memcpy activity at
   all — it is a CPU-only or fully idle window. Its feature vector is represented as an **empty
   dict**, not as a vector of zeros. An empty dict is a meaningful fingerprint in its own right:
   two empty-dict bins are identical (JS divergence = 0), and an empty-dict bin adjacent to any
   active bin has maximum divergence (JS = 1.0). Representing idleness as an empty dict rather
   than a zero vector also keeps the implementation clean — we never need to enumerate all
   vocabulary keys just to store zeros.

   **Note:** this per-bin empty-dict representation is used in Step 5a (boundary detection).
   At the *segment* level (Step 7), idle bins contribute an `"__idle__"` token rather than
   being silently skipped, so that GPU utilization is implicitly encoded in the segment
   distribution used by the DP merge step.

3. **Compute shares for active bins**: for each key in the fixed vocabulary, divide its raw time
   by `total_ns`:
   - Vocabulary kernels (top-`_TOP_K` by profile-wide GPU time): `share = bins[i].get(name, 0) / total_ns`
   - `"__memcpy__"`: `share = bins[i].get("__memcpy__", 0) / total_ns`
   - `"__other__"`: `share = sum of all bins[i] values for kernel names not in the top-K vocabulary / total_ns`

   The `"__other__"` entry exists because the vocabulary is finite. Any GPU time from kernels
   outside the top-`_TOP_K` is not discarded — it is lumped into a single bucket. This preserves
   the invariant that all shares sum to 1.0, which is required for JS divergence to be well-defined.
   The tradeoff is that two bins with different rare-kernel mixes but the same `"__other__"` share
   will appear identical to the divergence metric (see Limitations).

4. **Result**: a dict containing only keys with non-zero shares (sparse representation). All
   values sum to 1.0.

The output of this step is the list `dists[i]`: one `dict[str, float]` per bin (or an empty dict
for idle bins). These are the feature vectors used in all subsequent steps.

---

### Step 5 — Collect Candidate Boundaries

All candidates are tagged as `(timestamp_ns, priority)` pairs. Five sources contribute:

#### 5a — Distribution Divergence Peaks

For each adjacent pair of bins `(i, i+1)`, compute the **Jensen-Shannon divergence** between
`dists[i]` and `dists[i+1]`:

```
M[k] = (P[k] + Q[k]) / 2   for all k in the union of keys
JS(P, Q) = (KL(P‖M) + KL(Q‖M)) / 2
KL(P‖M) = Σ P[k] × log₂(P[k] / M[k])   for all k where P[k] > 0
```

Special cases that avoid numerical issues:
- Both dicts empty → JS = 0.0 (two idle bins are identical)
- One empty, one non-empty → JS = 1.0 (maximum divergence)

JS divergence is symmetric and bounded in [0, 1] (using log base 2). Sort all adjacent pairs by
JS divergence descending. Take the top `_N_CANDIDATES_FACTOR × max_phases` pairs. The boundary
timestamp for pair `(i, i+1)` is `t0 + (i + 1) × bin_width_ns` — the edge between the two bins.
These receive priority `_PRI_DIST`.

`max_phases` enters the algorithm here — and only here — as a control on how many distribution
boundary candidates are passed forward. Taking more candidates than needed wastes merge-step
iterations; taking too few risks missing a genuine boundary. The factor of `_N_CANDIDATES_FACTOR`
provides headroom above `max_phases` so that the pre-pass and greedy merge have material to work
with, while keeping the candidate count from growing without bound. Crucially, this does not
affect bin width and therefore does not affect boundary precision: a dominant phase transition
(such as the boundary between a short initialization phase and a long compute phase) will have
the highest JS divergence of any adjacent pair and will always be in the top candidates regardless
of what `max_phases` is set to.

Note: active↔idle transitions produce JS = 1.0 and will naturally appear at the top of this list.
Step 5b handles them explicitly to ensure they are always captured even if the candidate pool
would otherwise be too small.

#### 5b — Active/Idle Transition Boundaries

Scan `dists[]` for transitions between idle (empty dict) and non-idle (non-empty dict). For each
contiguous idle stretch spanning bins `[first_idle, last_idle]`, emit two candidates:

- **Start of idle stretch**: `t0 + first_idle_idx × bin_width_ns`  (priority `_PRI_DIST`)
- **End of idle stretch**: `t0 + (last_idle_idx + 1) × bin_width_ns`  (priority `_PRI_DIST`)

This guarantees that every GPU-idle or CPU-only period is bounded on both sides regardless of
whether its adjacent-bin divergence values ranked in the top-K from Step 5a.

#### 5c — Marker Range Boundaries (NVTX or rocTX)

Same logic as the current algorithm's `_nvtx_phase_boundaries()`. Collect the start and end
timestamps of top-level marker ranges (NVTX annotations on NVIDIA, rocTX ranges on AMD; ranges
not contained within any other long range in the set) that appear `≤ max_phases` times.
Frequently-repeated ranges (e.g., per-iteration annotations) are excluded from boundary
candidates because they mark iterations within a phase, not phase transitions — but they are
retained in a separate list for use in the labeling step (Step 10). These candidates receive
priority `_PRI_NVTX`.

#### 5d — MPI Barrier Cluster Ends

Same logic as the current algorithm's `_mpi_cluster_boundaries()`. Collect `MPI_Barrier` end
timestamps from `MPI_COLLECTIVES_EVENTS`, group into clusters separated by gaps of more than
500ms, and emit the end timestamp of each cluster. These candidates receive priority `_PRI_MPI`.

#### 5e — GPU Activity Edges

Query `MIN(start)` and `MAX(end)` from `CUPTI_ACTIVITY_KIND_KERNEL`. These mark the exact
timestamps when GPU kernel execution begins and ends and receive the highest priority,
`_PRI_GPU_EDGE`.

---

### Step 6 — Merge Nearby Candidates

Pool all tagged candidates from Step 5. Discard any candidate closer than `tolerance_ns` to `t0`
or `t1` (profile-edge candidates are not useful interior boundaries):

```
tolerance_ns = max(5_000_000,  int(span_ns × 0.005))   # 5ms or 0.5% of span
```

Sort the remaining candidates by timestamp. Walk them left to right, maintaining a list of accepted
boundaries. Each candidate is compared against the **last accepted boundary** (the current cluster
anchor):

- If the candidate is within `tolerance_ns` of the last accepted boundary: keep whichever has
  higher priority; ties retain the existing (earlier) entry.
- If the candidate is beyond `tolerance_ns` of the last accepted boundary: accept it
  unconditionally as the start of a new cluster.

Because the comparison is always against the last *accepted* boundary and not against every member
of the current cluster, the merge window does not slide. A chain of candidates where each
consecutive gap is below `tolerance_ns` but the total span exceeds `tolerance_ns` may produce more
than one surviving boundary: once the cumulative distance from the cluster anchor exceeds the
tolerance, the next candidate starts a fresh cluster regardless of how close it is to its immediate
predecessor.

No averaging or interpolation is performed — the surviving timestamp is always an actual event or
bin-edge timestamp.

Prepend `t0` and append `t1` to the surviving timestamps. The result is the final sorted boundary
list.

---

### Step 7 — Fingerprint Each Segment

For each interval `[boundaries[i], boundaries[i+1]]`, compute a `_Seg` containing:

- **`kernel_ns`**: total GPU kernel time within the window. Boundary-straddling kernels are
  clamped to the window edges and contribute partial time (same SQL as the current algorithm).
- **`dominant_kernel`**: the kernel **short name** with the most GPU time whose center
  `(start + end) / 2` falls inside the window (center-based attribution, same as current). Short
  name is used here — not demangled — because this field is only used for labeling (Step 10),
  where a compact, readable name is preferred over a full template instantiation string.
- **`distribution`**: the normalized distribution over the vocabulary for the segment as a whole.
  Computed by iterating over the bins whose center falls inside the segment window:
  - **Active bins** (non-empty): their kernel-name → nanoseconds entries are summed into a
    combined dict, exactly as in Step 4.
  - **Idle bins** (empty dict): each contributes `bin_width_ns` nanoseconds to the `"__idle__"`
    key in the combined dict.
  The combined dict is then normalized by its total to produce a probability distribution.
  This means `"__idle__"` weight ≈ fraction of bins in the segment that were GPU-idle, giving
  the distribution an implicit measure of GPU utilization. A segment with 1% utilization will
  have `"__idle__" ≈ 0.99`; a segment with 60% utilization will have `"__idle__" ≈ 0.40`. JS
  divergence between these two is high even if both run the same kernels — preventing the greedy
  merge from incorrectly treating a nearly-idle initialization phase as similar to an active
  compute phase. This is the key new field — it is what the merge criterion and pre-pass use.
- **`gpu_util`**: derived as `100 × kernel_ns / (end_ns - start_ns)` (unchanged).

---

### Step 8 — Pre-pass: Merge Identical Adjacent Segments

Scan adjacent segment pairs. If `JS(a.distribution, b.distribution) < _PREPASS_THRESHOLD`, merge
them unconditionally. When merging:

- `kernel_ns` is summed.
- `distribution` is recomputed as the **duration-weighted** average:
  `merged[k] = (a.distribution[k] × a.duration_ns + b.distribution[k] × b.duration_ns) / (a.duration_ns + b.duration_ns)`
  Duration weighting is correct here because the distribution now covers the full wall-clock window
  of each segment (active kernel time plus `"__idle__"` time), not just the active portion.
  Using `kernel_ns` as the weight would cause idle-heavy segments to contribute almost nothing
  to the merged distribution, discarding their utilization signal.
- `dominant_kernel` is inherited from whichever side had more `kernel_ns`.

Repeat the scan until a full pass produces no merges. This ensures that a long run of nearly
identical segments (e.g., thousands of CG iterations) collapses completely before the max_phases
limit comes into play.

`_PREPASS_THRESHOLD = 0.10` — segments whose JS divergence is below this are considered to have
the same bottleneck structure and no analytical benefit from being kept separate.

---

### Step 9 — Optimal k-Segmentation via Dynamic Programming

After the pre-pass, the remaining segments represent a compact set of candidate phases. This step
finds the **globally optimal** partition of those segments into `k` contiguous groups, where `k`
is chosen automatically from 1 to `max_phases` using the cost curve.

**Cost function.** For a group of consecutive segments `s[a], s[a+1], ..., s[b]`, the cost is
the duration-weighted sum of Jensen-Shannon divergences from each constituent's distribution to
the merged group distribution:

```
cost(a, b) = Σᵢ₌ₐᵇ  duration(sᵢ) × JS(sᵢ.distribution, D_merged)
```

where `D_merged` is the duration-weighted average distribution of all segments in the group
(computed by applying `_merge_segs` left-to-right). This is analogous to within-cluster inertia
in k-means, using JS divergence instead of squared Euclidean distance. A group of nearly
identical segments has cost ≈ 0; a group mixing a long idle segment with a high-utilization
compute segment has high cost.

**Dynamic programming recurrence.** Let `dp[k][i]` = minimum total cost of partitioning
`segs[0..i]` into exactly `k` groups. The recurrence is:

```
dp[1][i] = cost(0, i)
dp[k][i] = min_{m=k-1}^{i} ( dp[k-1][m-1] + cost(m, i) )   for k ≥ 2
```

This is the classic optimal sequence segmentation algorithm and runs in O(n²K) time, where n is
the number of pre-pass segments and K = min(max_phases, n). For typical values (n ≤ 15, K ≤ 10)
this is negligible. The cost matrix `C[a][b] = cost(a, b)` is precomputed in O(n³) time by
building each merged distribution incrementally left-to-right.

**k-selection.** Computing `dp[k][n-1]` for all k from 1 to K yields a cost curve that is
monotonically non-increasing. The algorithm selects the largest k where the marginal gain
justifies the additional phase:

```
total_range = cost(k=1) − cost(k=K)
marginal_gain(k) = (cost(k−1) − cost(k)) / total_range
best_k = largest k ≤ K where marginal_gain(k) ≥ _DP_ELBOW_THRESHOLD
```

`_DP_ELBOW_THRESHOLD = 0.05` means: add another phase only if doing so reduces the total cost
by at least 5% of the total possible improvement (from 1 phase to `max_phases` phases). If
`total_range == 0` (all segments are identical), `best_k = 1`.

**Why DP instead of greedy merge.** The greedy approach was locally optimal: at each step it
merged the single most similar adjacent pair, which could commit to a bad intermediate state that
prevented a globally better partition. The DP considers all possible k-partitions simultaneously
and picks the one that minimizes total heterogeneity — making it immune to this cascade failure.
In particular, a large idle segment will never be absorbed into an adjacent active segment during
the DP if keeping them separate produces a lower total cost, regardless of what happened to other
segments earlier in the search.

---

### Step 10 — Label and Deduplicate

Each final segment is labeled in priority order:

1. **Marker coverage**: for each marker range name (NVTX or rocTX; from the full list collected in
   Step 5c, including frequently-repeated ranges), sum the overlap of all ranges with that name
   against the segment window. If the best-covered name accounts for ≥ 30% of the segment's wall
   time, use that name as the label.
2. **Idle**: if no NVTX label applies and `gpu_util < _IDLE_LABEL_UTIL_THRESHOLD` (1%), label as
   `"idle"`. This threshold rather than a strict `kernel_ns == 0` check ensures that segments
   with a negligible trickle of GPU launches during what is effectively an MPI wait or
   synchronization barrier are not incorrectly labeled by their nominal dominant kernel.
3. **Dominant kernel**: if there is GPU activity but no sufficient NVTX coverage, use the
   dominant kernel short name.
4. **Unknown**: fallback if none of the above apply.

If multiple phases share the same label, append a counter suffix to disambiguate:
`"CG solver (1)"`, `"CG solver (2)"`, etc.

### Hard-Coded Values

| Constant | Value | What it controls | Sensitivity |
|---|---|---|---|
| `_TOP_K` | 20 | Kernel vocabulary size; kernels beyond this rank are bucketed as `"__other__"` | Low |
| `_N_BINS` | 500 | Target number of bins; `bin_width = span / _N_BINS` before floor/ceiling are applied. Controls boundary precision — independent of `max_phases`. | Medium |
| `_BIN_WIDTH_FLOOR_NS` | 50,000,000 (50 ms) | Minimum bin width; prevents overly fine bins on short profiles | Medium |
| `_BIN_WIDTH_CEIL_NS` | 500,000,000 (500 ms) | Maximum bin width; prevents overly coarse bins on very long profiles | Low |
| `_N_CANDIDATES_FACTOR` | 5 | Distribution boundary candidates = factor × max_phases; this is where max_phases influences boundary generation without affecting precision | Low |
| `_PREPASS_THRESHOLD` | 0.10 | JS divergence below this → unconditional pre-pass merge (same bottleneck) | High |
| `_DP_ELBOW_THRESHOLD` | 0.05 | Min fractional cost reduction (relative to total range) to justify one more phase in k-selection | High |
| `_PRI_DIST` | 2 | Priority of distribution-detected and idle-transition boundary candidates | Low |
| `_PRI_MPI` | 2 | Priority of MPI_Barrier cluster boundary candidates | Low |
| `_PRI_NVTX` | 3 | Priority of NVTX range boundary candidates | Low |
| `_PRI_GPU_EDGE` | 4 | Priority of GPU activity start/end boundary candidates | Low |
| merge tolerance | `max(5 ms, 0.5% of span)` | Window within which two boundary candidates are collapsed to one | Medium |
| NVTX min duration | `max(2% of span, 100 ms)` | Minimum NVTX range length to consider for boundaries or labeling | Low |
| MPI cluster gap | 500 ms | Gap between consecutive MPI_Barrier ends that starts a new cluster | Low |
| NVTX label threshold | 30% of segment wall time | Minimum NVTX coverage fraction to use a range name as the phase label | Low |

### Limitations

**1. Boundary precision is limited to one bin width, and short phases below one bin width are undetectable.**

Distribution-detected boundary timestamps are bin edges — artificial points computed as
`t0 + k × bin_width_ns` — not timestamps of real events. A true phase transition that occurs in
the middle of a bin will be detected at the nearest bin edge, introducing a position error of up
to one full bin width (up to 500 ms at the ceiling). In contrast, NVTX, MPI, and GPU-edge
boundary candidates all carry real event timestamps. When a distribution boundary and a real-event
boundary happen to be within merge tolerance of each other, the higher-priority real-event
timestamp wins — partially mitigating this error. But for transitions that produce no real-event
candidate (e.g., a gradual kernel-mix shift in the middle of sustained GPU activity), the bin-edge
precision is the best the algorithm can offer.

Additionally, any phase shorter than `bin_width_ns` (50 ms floor with `_N_BINS = 500`) will not
produce a distribution shift detectable from its neighbors, because its kernels will be spread
across adjacent bins. Such phases are silently absorbed into their neighbors.

*Future improvements to consider:*
- **Snap to nearest real event**: after a distribution boundary candidate is generated from a
  divergence peak, search within ±1 bin width for the nearest GPU idle gap edge, NVTX range
  boundary, or MPI barrier. If one is found within tolerance, replace the bin-edge timestamp with
  the real-event timestamp and upgrade its priority. This costs almost nothing and eliminates the
  precision error whenever a real event happens to be near the true boundary.
- **Two-pass adaptive refinement**: use a coarse first pass to identify high-divergence regions,
  then re-bin those regions at finer granularity to locate the boundary precisely. This achieves
  sub-bin precision without shrinking the global bin width (which would increase SQL work across
  the whole profile).

**2. CPU-only phases are indistinguishable from each other.**
Two CPU-only phases — for example, loading a gauge configuration and writing correlator output —
both produce empty feature vectors. They will always be merged into a single idle phase regardless
of their duration or semantic difference. NVTX annotations are the only remedy for this case.

**3. The `"__other__"` bucket can hide genuine structure.**
If a profile uses more than `_TOP_K` distinct kernels and the semantically important distinctions
fall outside the top-K by total GPU time, those distinctions are invisible to the JS divergence
metric. A phase dominated by rare-but-important kernels may appear identical to a phase with a
different rare-kernel mix, because both project to the same `"__other__"` share.

**4. Center-based attribution introduces a positioning error of up to one bin width.**
A kernel whose center falls just outside a bin is attributed to the neighboring bin. For typical
profiles where kernels are short relative to the bin width, this has negligible effect. For profiles
with very long individual kernels (duration comparable to `bin_width_ns`), boundary positions can
be off by up to one full bin.

**5. DP is globally optimal for the given pre-pass segments, but pre-pass quality still matters.**
The DP finds the best possible k-partition of whatever segments the pre-pass produces. If a
genuine phase boundary was missed entirely by the pre-pass (i.e., no candidate was generated for
it), the DP cannot recover it — no merge-based algorithm can split a segment that was never
created. The pre-pass uses a conservative threshold (`_PREPASS_THRESHOLD = 0.10`), so this
failure mode is rare, but it can occur for very subtle phase transitions where adjacent bins have
JS divergence below this threshold.

**6. JS divergence weights kernels by time share within active bins, not absolute time.**
A kernel that constitutes 60% of the active bins in a short segment has equal weight to one
constituting 60% of the active bins in a long segment. Similarity scores are therefore comparable
in the merge step regardless of segment length, which can cause short segments with unusual kernel
mixes to resist merging even when the absolute performance difference is negligible.

*Partially mitigated by the `"__idle__"` token (see Step 7):* segments with very different GPU
utilization levels will have different `"__idle__"` fractions and therefore non-trivial JS
divergence from each other, even if they run the same kernels in the same proportions. The
remaining limitation applies to pairs of segments that both have high (or both have low)
utilization but differ only in the absolute amount of kernel time.

**7. Profiles with no GPU kernel activity at all.**
If `CUPTI_ACTIVITY_KIND_KERNEL` is absent or empty, the vocabulary is empty and all feature
vectors are empty. The algorithm will return a single phase (or a small number of phases separated
only by MPI or NVTX candidates) regardless of `max_phases`. Memcpy-only profiles are partially
handled via the `"__memcpy__"` pseudo-kernel, but any phase distinctions that rely on kernel
distribution will be lost.

**8. No automated quality metric for the output.**
The algorithm has no way to assess whether the segmentation it produces is useful. It cannot
detect if `max_phases` is set too high (artificially split phases) or too low (over-merged phases
that should be separate). The `--verbose` flag exposes internal state for manual inspection, but
there is no quantitative score reported alongside the result.
